# Story 1.4 — Projetos Supabase Cloud e migration baseline

## Status

**Ready** — Validado pelo `@po` (Pax). GO 8.0/10. Liberado para `@dev`. ⚠️ T1 e T5 requerem ação manual do owner no dashboard Supabase antes da implementação.

---

## Story

**Como** dev mantenedor do youtube-biolink,
**eu quero** dois projetos Supabase na nuvem (`production` e `development`) e migration baseline versionada,
**para que** as próximas stories evoluam o schema a partir de uma base controlada, sem dependência de instância local.

---

## Acceptance Criteria

> Fonte canônica: `docs/prd.md` § 6 — Epic 1, Story 1.4.

1. Dois projetos criados na conta Supabase do owner: `youtube-biolink-prod` e `youtube-biolink-dev` (região `sa-east-1` se disponível, senão `us-east-1`).
2. Supabase CLI instalada localmente; `supabase init` executado; `supabase/config.toml` commitado.
3. `supabase link` configurado por ambiente — variável `SUPABASE_PROJECT_REF` no `.env.local` aponta para o projeto `development`; secret no CI/Vercel aponta para `production`.
4. Migration baseline `supabase/migrations/{ts}_baseline.sql` cria extensões necessárias (`pgcrypto`, `citext`) e função `set_updated_at()`.
5. `supabase db push` aplica baseline com sucesso tanto no projeto `development` quanto `production`.
6. `.env.example` lista `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_PROJECT_REF` com placeholders e instruções. **`SUPABASE_SERVICE_ROLE_KEY` não entra no MVP** (entra no Epic 6 com RLS).
7. README da raiz atualizado com seção "Setup local" explicando o fluxo: criar conta Supabase → obter credenciais do projeto `development` → preencher `.env.local` → `pnpm install` → `pnpm dev`.

## Dev Notes

> **CRITICAL:** Toda informação técnica abaixo veio de `docs/architecture.md` e `docs/prd.md` (citados). Nada inventado.

### Dois Projetos Supabase (sem instância local)

**`production`:** usado por Vercel deploy de `main`; credenciais como secrets Vercel.
**`development`:** usado por devs locais + integration tests CI. [Source: architecture.md § 2.2]

Não há Supabase local em momento algum do projeto. [Source: architecture.md § 2.2 — "Supabase Cloud only"]

### SQL da Migration Baseline

Conteúdo exato em [Source: architecture.md § 9.1]:
```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "citext";

CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;
```

- `pgcrypto`: fornece `gen_random_uuid()` usado em tabelas futuras.
- `citext`: tipo case-insensitive para `username` (Stories 2.x).
- `set_updated_at()`: trigger reutilizado por todas as tabelas com `updated_at`.

### Variáveis de Ambiente

```
NEXT_PUBLIC_SUPABASE_URL       # URL pública do projeto (exposta ao client)
NEXT_PUBLIC_SUPABASE_ANON_KEY  # chave anon (exposta ao client, segura sem RLS no MVP)
SUPABASE_PROJECT_REF           # ref do projeto (para supabase link)
```

`SUPABASE_SERVICE_ROLE_KEY` **não entra no MVP** — introduzido no Epic 6 junto com RLS. Disponível como secret CI para integration test cleanup (deletar usuários de teste). [Source: architecture.md § 13.2 nota]

### Clients Supabase no Next.js

`lib/supabase.ts` (a criar em stories futuras) usa `@supabase/ssr` para gerenciar cookies HttpOnly. [Source: architecture.md § 11.2]

### Estrutura de Arquivos Supabase

```
supabase/
├── config.toml
└── migrations/
    └── {ts}_baseline.sql
```
[Source: architecture.md § 12]
