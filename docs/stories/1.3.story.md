```
│   └── components/            # (future) RTL component tests
└── integration/               # contra Supabase development
    └── helpers/
        ├── unique-user.ts     # (placeholder — implementar nas stories de feature)
        └── cleanup.ts         # (placeholder)
```
[Source: architecture.md § 15.2]

### Estratégia de Isolamento de Integration Tests

Cada integration test cria usuário próprio (`test-{runId}-{uuid}@youtube-biolink.test`), opera com suas credenciais, e deleta no teardown via `admin.auth.admin.deleteUser()`. [Source: architecture.md § 15.3]

- `SUPABASE_SERVICE_ROLE_KEY` é necessário para o helper de cleanup — disponível em CI como secret.
- Sem RLS no MVP, não há testes de RLS cross-user nesta fase.
- CI serializa integration tests via `concurrency.group: integration-tests-development`. [Source: architecture.md § 13.2]

### Nota sobre `@/*` alias no Vitest

O `vitest.config.ts` replica o alias `@/*` do `tsconfig.json` usando `path.resolve(__dirname, '.')` como raiz.

### Decisão: Config Simples vs Workspace API

A workspace API do Vitest 4.x foi movida para `vitest.workspace.ts` (arquivo separado). Para simplicidade e compatibilidade, utilizou-se config simples com `include: ['tests/**/*.test.{ts,tsx}']` e `exclude` da pasta `.aiox-core`. Os scripts `test:unit` e `test:integration` filtram por diretório via argumento CLI com `--passWithNoTests` para CI-readiness quando as pastas ainda estão vazias.

### Scripts no package.json

```json
"test": "vitest run",
"test:watch": "vitest",
"test:unit": "vitest run --reporter=verbose --passWithNoTests tests/unit",
"test:integration": "vitest run --reporter=verbose --passWithNoTests tests/integration"
```

Story 1.5 (CI) rodará `pnpm test:unit` (sem secrets) e `pnpm test:integration` (com secrets `SUPABASE_URL_DEV` + `SUPABASE_ANON_KEY_DEV`). [Source: architecture.md § 13.2]
