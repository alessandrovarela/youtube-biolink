### Dois Projetos Supabase (sem instância local)

**`production`:** usado por Vercel deploy de `main`; credenciais como secrets Vercel.
**`development`:** usado por devs locais + integration tests CI. [Source: architecture.md § 2.2]

Não há Supabase local em momento algum do projeto. [Source: architecture.md § 2.2 — "Supabase Cloud only"]

### SQL da Migration Baseline

Conteúdo exato em [Source: architecture.md § 9.1]:
```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "citext";

CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;
```

- `pgcrypto`: fornece `gen_random_uuid()` usado em tabelas futuras.
- `citext`: tipo case-insensitive para `username` (Stories 2.x).
- `set_updated_at()`: trigger reutilizado por todas as tabelas com `updated_at`.

### Variáveis de Ambiente

```
NEXT_PUBLIC_SUPABASE_URL       # URL pública do projeto (exposta ao client)
NEXT_PUBLIC_SUPABASE_ANON_KEY  # chave anon (exposta ao client, segura sem RLS no MVP)
SUPABASE_PROJECT_REF           # ref do projeto (para supabase link)
```

`SUPABASE_SERVICE_ROLE_KEY` **não entra no MVP** — introduzido no Epic 6 junto com RLS. Disponível como secret CI para integration test cleanup (deletar usuários de teste). [Source: architecture.md § 13.2 nota]

### Clients Supabase no Next.js

`lib/supabase.ts` (a criar em stories futuras) usa `@supabase/ssr` para gerenciar cookies HttpOnly. [Source: architecture.md § 11.2]

### Estrutura de Arquivos Supabase

```
supabase/
├── config.toml
└── migrations/
    └── {ts}_baseline.sql
```
[Source: architecture.md § 12]
