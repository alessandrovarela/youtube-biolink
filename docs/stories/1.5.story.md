# Story 1.5 — GitHub Actions CI

## Status

**Ready** — Validado pelo `@po` (Pax). GO 8.5/10. Liberado para `@dev`. ⚠️ T4 (branch protection) é responsabilidade de @devops/owner, não @dev.

---

## Story

**Como** dev mantenedor do youtube-biolink,
**eu quero** um workflow de CI que rode typecheck + lint + test + build em todo PR,
**para que** regressões sejam bloqueadas antes do merge e o repositório mantenha qualidade contínua.

---

## Acceptance Criteria

> Fonte canônica: `docs/prd.md` § 6 — Epic 1, Story 1.5.

1. Workflow `.github/workflows/ci.yml` dispara em `pull_request` e `push` para `main`.
2. Jobs executam, em ordem: `pnpm install --frozen-lockfile` → `pnpm typecheck` → `pnpm lint` → `pnpm test:unit` → `pnpm test:integration` → `pnpm build`.
3. Integration tests usam secrets `SUPABASE_URL_DEV` e `SUPABASE_ANON_KEY_DEV` configurados em GitHub Secrets; unit tests não requerem credenciais.
4. Cache de pnpm e Next.js configurado para reduzir tempo de execução.
5. CI roda em <5min no estado atual do projeto.
6. Branch protection na `main` habilitada exigindo CI verde antes do merge.

## Tasks / Subtasks

- [ ] **T1. Criar `.github/workflows/ci.yml`** (AC: 1, 2, 3, 4)
  - [ ] Criar o arquivo com conteúdo baseado no esboço da Architecture [Source: architecture.md § 13.2]:
    ```yaml
    name: CI
    on:
      pull_request:
      push:
        branches: [main]

    concurrency:
      group: ${{ github.workflow }}-${{ github.ref }}
      cancel-in-progress: true

    jobs:
      quality:
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - uses: pnpm/action-setup@v4
            with:
              version: 9

          - uses: actions/setup-node@v4
            with:
              node-version: 20
              cache: pnpm

          - name: Install dependencies
            run: pnpm install --frozen-lockfile

          - name: Typecheck
            run: pnpm typecheck

          - name: Lint
            run: pnpm lint

          - name: Unit tests
            run: pnpm test:unit

          - name: Integration tests
            run: pnpm test:integration
            env:
              NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.SUPABASE_URL_DEV }}
              NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.SUPABASE_ANON_KEY_DEV }}

          - name: Build
            run: pnpm build
    ```
  - [ ] Nota: `concurrency` com `cancel-in-progress: true` cancela runs antigas do mesmo branch em PRs.
  - [ ] Para integration tests, o `concurrency.group` de § 13.2 usa `integration-tests-development` para serializar — avaliar se necessário no estado atual (sem integration tests reais ainda; ajustar em Stories futuras).

- [ ] **T2. Cache de pnpm e Next.js** (AC: 4)
  - [ ] `pnpm/action-setup@v4` com `version: 9` já habilita cache automático de pnpm com `actions/setup-node`.
  - [ ] Adicionar cache do `.next/cache` para builds incrementais:

    ```yaml
    - name: Cache Next.js build

          ${{ runner.os }}-nextjs-${{ hashFiles('pnpm-lock.yaml') }}-
    ```

## Dev Notes

> **CRITICAL:** Toda informação técnica abaixo veio de `docs/architecture.md` e `docs/prd.md` (citados). Nada inventado.

### CI Pipeline (esboço Architecture)

```yaml
name: CI
on:
  pull_request:
  push: { branches: [main] }

concurrency:
  group: integration-tests-development
  cancel-in-progress: false
```
[Source: architecture.md § 13.2]

Nota: o `concurrency.group` fixo em `integration-tests-development` **serializa** todos os runs de integration test para evitar conflito no banco `development`. Considerar usar `${{ github.workflow }}-${{ github.ref }}` para PRs e o grupo fixo apenas para `test:integration`. Avaliar na implementação.

### Versões de Ferramentas no CI

- Node.js: 20.x (LTS) [Source: architecture.md § 3.1]
- pnpm: 9.x [Source: architecture.md § 3.1]
- `actions/checkout@v4`, `pnpm/action-setup@v4`, `actions/setup-node@v4` (current stable)

### Scripts Esperados

```
pnpm typecheck      → tsc --noEmit
pnpm lint           → eslint
pnpm test:unit      → vitest run --project unit
pnpm test:integration → vitest run --project integration
pnpm build          → next build
```

Stories 1.2 e 1.3 criam esses scripts. Esta story assume que estão presentes.

### Secrets no CI

`SUPABASE_URL_DEV` e `SUPABASE_ANON_KEY_DEV` são usados apenas em `test:integration`. `SUPABASE_SERVICE_ROLE_KEY` fica disponível como secret para o helper de cleanup de integration tests (`admin.auth.admin.deleteUser()`), mesmo que não seja usado em Server Actions de produção. [Source: architecture.md § 13.2 nota]

### Estrutura de Arquivos

```
.github/
└── workflows/
    └── ci.yml
```
[Source: architecture.md § 12]
